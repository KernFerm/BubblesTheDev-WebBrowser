# Transparency Checklist

## What This Feature Does

This checklist helps people review whether the browser is being explained in a clear, trustworthy way.

It is useful for:

* project maintainers
* curious users
* developers
* security-conscious visitors

## Why It Matters

People should not have to guess what a privacy-focused browser is promising.

This checklist helps keep public claims grounded in documented behavior instead of hype.

## Privacy Impact

The checklist encourages public claims that match the documented local-first model.

It pushes the project toward statements like:

* browser data stays local by default
* diagnostics are user-controlled
* AI features are designed to stay local where supported
* profiles are isolated

It also discourages exaggerated claims that can damage trust.

## Accessibility Impact

Clear trust language helps accessibility users because plain explanations reduce ambiguity and cognitive load.

## Example Workflow

A parent is thinking about installing the browser for a family computer.

They review the checklist and confirm that the project explains:

1. what data stays local
2. what can use the network
3. how accessibility is supported
4. how diagnostics are controlled
5. what kinds of claims should be avoided

That helps them make a more informed decision.

## Technical Notes

Use this checklist when updating docs, release notes, examples, or public repository pages.

Public trust signals should include:

* published privacy documentation
* published security documentation
* plain-English transparency examples
* realistic feature descriptions
* no real secrets in public materials

Claims to avoid include:

* "the browser never connects to the internet"
* "the browser collects absolutely nothing under any circumstance"
* "the browser is impossible to exploit"

Those claims are too broad for a real browser platform.
